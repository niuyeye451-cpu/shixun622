## 一、具体模型列表

本系统共涉及 **3个预训练深度学习模型 + 1个大语言模型 + 2个自研算法引擎**，总计6个核心模型/算法：

### 1.1 预训练深度学习模型（3个）

| # | 模型名称 | 模型标识（HuggingFace） | 参数量 | 用途 | 选型理由 |
|---|---------|----------------------|--------|------|---------|
| 1 | **BioBERT** | `dmis-lab/biobert-v1.1` | 110M | 医疗实体识别（NER），6类实体 | 在PubMed+PMC生物医学文献上预训练，对医学术语理解远超通用BERT，F1可达0.92 |
| 2 | **RoBERTa** | `roberta-base` | 125M | 意图分类，5类意图 | 相比BERT有更优的微调性能，准确率目标≥92% |
| 3 | **Sentence-BERT** | `paraphrase-multilingual-MiniLM-L12-v2` | ~118M | 文本向量化（Embedding），384维向量 | 轻量级，支持中文，GPU友好，用于RAG向量检索的文本编码 |

### 1.2 大语言模型（1个）

| # | 模型名称 | 部署方式 | 用途 | 说明 |
|---|---------|---------|------|------|
| 4 | **DeepSeek / Qwen** | API调用或本地GPU部署 | RAG答案生成（LLM） | 原型阶段用规则模板拼装答案，生产阶段接入LLM进行自然语言生成 |

### 1.3 自研算法引擎（2个）

| # | 算法名称 | 实现方式 | 用途 | 核心逻辑 |

## 二、研究过程与思路

### 总体研究路线

```
技术调研 → 模型选型对比 → 架构设计 → 原型验证 → 评估数据集构建 → 指标验收
```

我们采用 **"渐进式升级"** 策略：先用规则引擎快速出原型验证架构可行性，再逐步替换为深度学习模型提升精度。这样做的优势是：

- **不依赖外部API即可演示**：规则模板可独立运行
- **架构稳定性高**：无论底层模型如何替换，上层架构不变
- **成本可控**：原型阶段不产生API调用费用

### 2.1 NLU 自然语言理解模块

#### 研究思路

```
用户输入 → 实体识别(NER) → 同义词归一化 → 意图分类 → 结构化NLU结果
```

#### 实体识别（NER）— 模型选型对比

| 方法 | F1值 | 优缺点 | 选择 |
|------|------|--------|------|
| Aho-Corasick规则匹配 + 医学词典 | ~0.75 | 快速原型、零依赖，但覆盖率低 | **原型阶段使用** |
| BiLSTM-CRF | ~0.85 | 传统深度学习方法，需要大量标注数据 | 不选 |
| **BioBERT-base** | **~0.92** | PubMed+PMC预训练，6M+生物医学摘要语料，医学实体识别领先 | **目标阶段采用** |

**选型结论**：选择BioBERT是因为它在生物医学领域的预训练使其天然理解医学实体边界。对于中文场景，可结合PengCheng-Medical-BERT等中文医学预训练模型进行微调。

#### 实体类型定义（6类BIO标注体系）

```
B-DIS I-DIS I-DIS  O    O    B-SYM I-SYM
高    血    压    患   者   头    痛

实体类型: Disease(疾病) / Drug(药品) / Symptom(症状) / Examination(检查) / Department(科室) / Population(人群)
```

#### 意图分类 — 5类意图体系

| 意图标识 | 中文含义 | 典型问句 | 触发特征词 |
|---------|---------|---------|-----------|
| `consult_symptom` | 症状咨询 | "头晕恶心是什么原因" | 怎么办、怎么治、严重吗 |
| `consult_drug` | 用药咨询 | "布洛芬一天最多吃几次" | 怎么吃、用量、副作用 |
| `disease_query` | 疾病查询 | "糖尿病的早期症状有哪些" | 什么是、症状、病因 |
| `contraindication_query` | 禁忌查询 | "高血压不能吃什么药" | 禁忌、不能吃、不宜 |
| `dept_recommend` | 科室推荐 | "胸闷应该挂什么科" | 挂什么科、看什么、哪个科 |

**选型结论**：原型阶段用正则规则匹配，生产阶段用RoBERTa-base（125M参数）微调5分类器，目标准确率≥92%。

#### 同义词归一化

```
用户口语  →  标准医学术语
"拉肚子"  →  "腹泻"
"脑壳疼"  →  "头痛"
"血压高"  →  "高血压"
"拍片"    →  "X线摄影"
```

基于Redis缓存实现高速查询，目标覆盖≥3000条映射。

### 2.2 Text2Cypher 自然语言转图查询模块

#### 研究思路

采用 **"模板匹配为主（覆盖率≥80%）+ 深度学习为辅（复杂查询）"** 的双策略：

| 策略 | 覆盖率 | 准确率 | 适用 | 成熟度 |
|------|--------|--------|------|--------|
| 预定义Cypher模板匹配 | 80%+ | 100%（确定性） | 5类预设意图 | **当前采用** |
| 深度学习（CodeLlama等） | 补充20% | 模型依赖（~69%） | 复杂多跳查询 | 未来升级方向 |

#### 预定义Cypher模板（≥10种）

```cypher
-- 模板1：症状查疾病
MATCH (d:Disease)-[:HAS_SYMPTOM]->(s:Symptom {name: $symptom})
RETURN d.name, d.description

-- 模板2：疾病查药品
MATCH (d:Disease {name: $disease})<-[:TREATS]-(drug:Drug)
RETURN drug.name, drug.usage

-- 模板3：药品查禁忌
MATCH (drug:Drug {name: $drug})-[:CONTRAINDICATED_FOR]->(d:Disease)
RETURN d.name AS contraindicated_condition

-- 模板4：症状查科室
MATCH (s:Symptom)<-[:HAS_SYMPTOM]-(d:Disease)-[:BELONGS_TO]->(dept:Department)
WHERE s.name IN $symptoms
RETURN dept.name, dept.description

-- 模板5：药品相互作用
MATCH (d1:Drug {name: $drug1})-[r:INTERACTS_WITH|CONTRAINDICATED_WITH|CAUTION_WITH]-(d2:Drug {name: $drug2})
RETURN d1.name, d2.name, type(r) AS rel_type
```

#### 安全沙箱设计（四层防护）

这是Text2Cypher模块的核心安全设计：

```python
FORBIDDEN_KEYWORDS = ["create", "delete", "set", "merge", "remove", "drop"]  # 第1层：禁止写操作
MAX_QUERY_DEPTH = 5      # 第2层：图遍历深度上限
MAX_RESULT_COUNT = 200   # 第3层：返回结果上限
QUERY_TIMEOUT_SEC = 3    # 第4层：查询超时熔断
```

#### 前沿论文参考

调研了2024年SyntheT2C/MedT2C论文（arxiv 2406.10710）：使用LLM提示生成+模板填充双管道合成Text2Cypher训练数据，CodeLlama-13B在Hetionet医疗数据库上达到69.2%执行准确率，接近GPT-4o的72.1%。这作为我们未来升级深度学习方法的技术储备。

### 2.3 RAG 检索增强生成模块

#### 研究思路 — 混合检索架构

我们的RAG模块不是传统的"向量检索→LLM生成"简单流水线，而是 **图数据库结构查询 + 向量语义检索 双引擎混合架构**：

```
用户问题
   │
   ├─→ Text2Cypher → Neo4j图数据库（结构化知识：实体/关系/三元组）
   │
   ├─→ Sentence-BERT → Milvus向量库（非结构化知识：指南/文献/说明书）
   │
   └─→ Prompt组装 → LLM生成 → 置信度评估 → 答案校验 → 返回用户
```

**为什么用混合检索？** 技术调研数据显示混合检索召回率91.32%，远超纯向量检索70.12%和纯BM25 61.08%。

#### 置信度量化公式（核心研究成果）

$$S_{final} = 0.2 \cdot Sim_{Milvus} + 0.2 \cdot Rel_{Neo4j} + 0.4 \cdot Faithfulness_{Ragas} + 0.2 \cdot AnswerRelevance_{Ragas}$$

**权重设计逻辑**：

| 维度 | 权重 | 说明 |
|------|------|------|
| 向量相似度 (Sim_Milvus) | 20% | 检索质量的客观指标 |
| 图谱关联度 (Rel_Neo4j) | 20% | 结构化知识的匹配度 |
| **忠实度 (Faithfulness)** | **40%** | **权重最高**——医疗场景下答案必须忠实于知识源，"宁可不说，不可乱说" |
| 答案相关性 (AnswerRelevance) | 20% | 端到端质量评估 |

**兜底机制**：当 S_final < 0.95 时，**强制触发安全兜底**，输出统一话术：
> "当前知识库暂无确切医学依据，为保障您的用药及健康安全，建议您及时前往医院线下就诊，遵医嘱治疗。"

#### RAG架构的医疗特化设计（4项差异化设计）

| 设计点 | 说明 | 违反后果 |
|--------|------|---------|
| **免责声明强制拼接** | 所有答案末尾必须拼接医疗免责声明 | 安全红线 |
| **禁忌症100%包含** | 处方药品信息必须包含禁忌症+不良反应 | 安全红线 |
| **来源强制标注** | 每条结论明确标注出处（指南/药典/文献） | 安全红线 |
| **拒绝编造能力** | 虚构药物/超范围问题直接兜底，不编造答案 | 安全红线 |

**前沿论文参考**：

- RAG² (NAACL 2025 Oral)：合理性引导的医学RAG，准确率提升6.1%
- GraphRAG for Rare Genetic Disease (npj Digital Medicine 2025)：Cypher RAG + Vector RAG临床验证，幻觉变异度降低53.94%
- VeReaFine (BioNLP 2025)：迭代验证精炼，多跳推理错误减少26%

### 2.4 知识推理模块

#### 研究思路 — BFS多跳路径查找

当用户问"高血压和肾功能衰竭有什么关系？"，系统需要在知识图谱中寻找多跳路径：

```
高血压 →[并发症]→ 糖尿病肾病 →[导致]→ 肾功能衰竭    （3跳）
高血压 →[用药]→ ACEI类药物 →[不良反应]→ 肾功能损伤   （3跳）
```

#### 算法：Neo4j BFS广度优先遍历

```cypher
MATCH p=(a {name: $source})-[*1..5]-(b {name: $target})
RETURN 
  [node IN nodes(p) | node.name] AS path_nodes,
  [rel IN relationships(p) | type(rel)] AS path_relations,
  length(p) AS hops
ORDER BY hops ASC
LIMIT 5
```

#### 路径评分公式

$$confidence(path) = \frac{1.0}{1.0 + 0.3 \times hops}$$

| 跳数 | 置信度 | 示例 |
|------|--------|------|
| 1跳 | 1.0 / 1.3 ≈ 0.769 | A直接连B |
| 2跳 | 1.0 / 1.6 ≈ 0.625 | A→X→B |
| 3跳 | 1.0 / 1.9 ≈ 0.526 | A→X→Y→B |
| 5跳 | 1.0 / 2.5 ≈ 0.400 | 最长允许路径 |

**设计逻辑**：路径越短，实体间关系越直接，可信度越高。

### 2.5 药物相互作用检测模块

#### 研究思路 — 多药两两交叉比对 + 三级严重性分级

```
用户输入: ["华法林", "阿司匹林", "布洛芬"]
         ↓
    两两比对:
    华法林 ←→ 阿司匹林  → RED 禁忌联用（严重出血风险）
    华法林 ←→ 布洛芬    → ORANGE 谨慎联用
    阿司匹林 ←→ 布洛芬  → YELLOW 潜在注意
```

#### 三级严重性分级体系

| 级别 | 颜色 | 标签 | 关系类型（Neo4j） | UI动作 | 示例 |
|------|------|------|------------------|--------|------|
| **一级** | 🔴红色 | 禁忌联用 | `CONTRAINDICATED_WITH` | 强遮罩阻断对话框 | 华法林+阿司匹林 |
| **二级** | 🟠橙色 | 谨慎联用 | `CAUTION_WITH` | 橙色卡片提示 | ACEI+保钾利尿剂 |
| **三级** | 🟡黄色 | 潜在注意 | `INTERACTS_WITH` | 黄色说明卡片 | 布洛芬+奥美拉唑 |

#### 核心Cypher查询

```cypher
MATCH (d1:Drug)-[r:CONTRAINDICATED_WITH|CAUTION_WITH|INTERACTS_WITH]-(d2:Drug)
WHERE d1.name IN $drugs AND d2.name IN $drugs AND id(d1) < id(d2)
RETURN d1.name AS drug_a, d2.name AS drug_b, type(r) AS rel_type
```

### 2.6 答案校验模块

#### 研究思路 — 最后一道安全防线的四维校验

```
RAG生成答案 → 四维校验器 → 通过 → 追加免责声明 → 返回用户
                         → 拦截 → 返回安全兜底话术
```

#### 四维校验体系

| 维度 | 检查内容 | 实现方法 | 红线 |
|------|---------|---------|------|
| **实体一致性** | 答案中的医学实体是否存在于知识源 | BioBERT提取答案实体，与源实体做相似度匹配 | 匹配阈值≥0.95 |
| **来源标注** | 每条结论是否标注出处 | 正则检查"来源/出处/参考文献/指南/药典"关键词 | **100%** |
| **安全性拦截** | 是否包含危险医疗建议 | 30条有害建议模式匹配 | **零漏过** |
| **完整性** | 答案结构是否完整 | 检查"核心结论→详细说明→注意事项→知识来源"四段结构 | 遗漏率≤20% |

#### 安全性拦截规则

```python
DANGER_PATTERNS = [
    (r'每日\s*\d+\s*片',    '明确剂量建议'),       # 禁止给具体剂量
    (r'可以治愈',           '声称治愈'),            # 医学不可承诺100%
    (r'保证|100%|绝对',     '保证性用语'),          # 不能使用绝对化用语
]
```

## 四、安全零容忍设计

本项目在医疗场景下设定了 **6项安全零容忍项**，任一项FAIL则系统整体不通过：

| # | 零容忍项 | 所属模块 | 逻辑 | 验收方法 |
|---|---------|---------|------|---------|
| 1 | **置信度<95%必须触发兜底** | RAG | 不可输出不可靠的医疗建议 | UNK-30测试集，S_final<0.95时100%触发 |
| 2 | **RED级药物相互作用不得漏检** | DDI检测 | 禁忌联用漏检可能致命 | DDI-50测试集，RED级别全部检出 |
| 3 | **有害答案必须拦截** | 答案校验 | 不可输出超剂量/假疗法/错误禁忌 | SAF-30测试集，漏过=0 |
| 4 | **来源标注率必须100%** | 答案校验 | 每条结论必须有出处可追溯 | 正则检查所有输出 |
| 5 | **处方药品必须100%包含禁忌症+不良反应** | RAG | 用药安全底线 | QA-100中药品咨询部分的完整性检查 |
| 6 | **Cypher安全沙箱零漏过** | Text2Cypher | 防止恶意注入/数据破坏 | 专项注入测试8项 |

## 五、容错降级设计

当AI服务出现故障时，系统具备自动降级能力（9种场景全覆盖）：

| # | 故障场景 | 降级策略 |
|---|---------|---------|
| 1 | NLU微服务超时(>200ms) | 降级为关键词规则引擎，confidence=0.5 |
| 2 | NLU微服务宕机 | 同上 |
| 3 | Neo4j连接失败(5s内连续10次) | CircuitBreaker熔断，标记Open状态，不再透传 |
| 4 | Milvus不可用 | 仅用Neo4j结果生成答案，confidence降低 |
| 5 | LLM API超时 | 指数退避重试(1s→2s→4s→8s→16s)，5次全失败→基础模板查询 |
| 6 | LLM API大规模故障 | 基础模式 + 黄底黑字提示条 |
| 7 | Redis不可用 | 多轮对话上下文丢失，5轮历史失效 |
| 8 | 降级模式UI | 前台页面顶部黄底黑字滚动提示条 |
| 9 | 故障恢复 | 自动检测数据库可用性，恢复后自动退出降级 |

## 六、研究结论

### 6.1 核心结论

1. **模型选型有据可查**：BioBERT（NER）、RoBERTa（意图）、Sentence-BERT（Embedding）、DeepSeek/Qwen（LLM生成）四个深度学习模型的选择均基于技术调研和对比实验数据（规则→BiLSTM→BioBERT的渐进路线）。

2. **评估体系完整**：设计了8个标准化测试数据集 + 23项A级红线指标 + 6项安全零容忍 + 9种容错降级场景，覆盖全部6个AI模块的功能/性能/安全维度。

3. **安全设计多层**：置信度兜底 + 四维答案校验 + 安全沙箱 + 免责声明 + 容错降级，五层安全防护。

4. **混合检索创新**：图数据库+向量库双引擎RAG架构，混合检索召回率91.32%（vs纯向量70.12%），置信度加权公式有明确的权重设计逻辑。

5. **工程化可落地**：渐进式升级策略（规则→深度学习→LLM），确保任何时候都有可演示的系统版本。
