## 1. 医疗知识图谱构建

### 1.1 构建流程全景

医疗知识图谱的标准构建流水线：

```
数据采集 → 实体识别(NER) → 关系抽取(RE) → 知识融合 → Neo4j存储 → 图谱应用
```

### 1.2 核心实体与关系类型

参考业界最佳实践，医疗知识图谱的标准Schema：

**实体类型（8+类）**：
| 实体类型 | 示例 | 颜色标识（本项目规范） |
|---------|------|---------------------|
| Disease（疾病） | 2型糖尿病、高血压 | 蓝色 #165DFF |
| Drug（药品） | 二甲双胍、复方降压片 | 绿色 #00B42A |
| Symptom（症状） | 发热、头痛、咳嗽 | 红色 #F53F3F |
| Department（科室） | 心内科、内分泌科 | 橙色 #FF7D00 |
| Examination（检查） | 血糖检测、心电图 | 紫色 #722ED1 |
| Population（人群） | 孕妇、老年人、儿童 | — |
| Food（食物） | 高蛋白食物 | — |
| Producer（生产厂商） | — | — |

**关系类型（11+类）**：
| 关系 | 头实体→尾实体 | 语义 |
|------|-------------|------|
| TREATS / TREAT_BY | 药品→疾病 | 治疗 |
| HAS_SYMPTOM | 疾病→症状 | 表现为 |
| CAUSES | 药品→不良反应 | 引起 |
| CONTRAINDICATED_FOR / CONTRAINDICATED_WITH | 药品→疾病/药品 | 禁忌 |
| BELONGS_TO | 疾病→科室 | 所属科室 |
| NEED_CHECK / EXAMINES | 疾病→检查 | 诊断依据 |
| COMPLICATION / ACOMPANY_WITH | 疾病→疾病 | 并发症 |
| DO_EAT / NOT_EAT | 疾病→食物 | 宜吃/忌吃 |
| APPLICABLE_TO | 药品→人群 | 适用于 |
| INTERACTS_WITH | 药品→药品 | 相互作用 |
| CAUTION_WITH | 药品→药品 | 谨慎联用 |

### 1.3 Neo4j 最佳实践

**索引策略**：
```cypher
CREATE INDEX ON :Disease(name);
CREATE INDEX ON :Disease(icd10);
CREATE INDEX ON :Symptom(name);
CREATE INDEX ON :Drug(name);
CREATE INDEX ON :Department(name);
```

**性能配置**（针对10万实体+50万关系规模）：
- `dbms.memory.heap.max_size=24G`（堆内存设为系统内存的50%）
- `dbms.memory.pagecache.size=10G`（页面缓存）
- 使用 `neo4j-admin import` 批量导入CSV数据

**查询优化要点**：
- 限制查询深度≤5跳
- 限制返回结果≤200条（项目需求书硬性要求）
- 热门查询结果缓存至Redis（TTL 1小时）
- 对固定模式路径考虑物化视图

**部署架构**：
- 三节点主从集群（Primary负责写入/版本发布，Replica承载Cypher读查询）
- Raft协议故障转移

## 3. Text2Cypher自然语言转图查询

### 3.1 技术路线

根据项目需求书，采用**模板匹配为主（覆盖率≥80%）+ 深度学习为辅**的双策略：

| 策略 | 覆盖率 | 适用场景 | 准确率 |
|------|--------|---------|--------|
| 模板匹配 | 80%+ | 预设意图类型（≥10种模板） | 100%（确定性） |
| 深度学习生成 | 20% | 复杂多跳查询 | 模型依赖 |

### 3.2 预定义Cypher模板（≥10种）

```cypher
-- 模板1：症状查疾病
MATCH (d:Disease)-[:HAS_SYMPTOM]->(s:Symptom {name: $symptom})
RETURN d.name, d.description

-- 模板2：疾病查药品
MATCH (d:Disease {name: $disease})-[:TREATED_BY]->(drug:Drug)
RETURN drug.name, drug.usage, drug.dosage

-- 模板3：药品查禁忌
MATCH (drug:Drug {name: $drug})-[:CONTRAINDICATED_FOR]->(d:Disease)
RETURN d.name AS contraindicated_condition

-- 模板4：症状查科室
MATCH (s:Symptom)-[:BELONGS_TO]->(d:Disease)-[:TREATED_BY]->(dept:Department)
WHERE s.name IN $symptoms
RETURN dept.name, dept.description

-- 模板5：药品相互作用
MATCH (d1:Drug {name: $drug1})-[r:INTERACTS_WITH|CONTRAINDICATED_WITH|CAUTION_WITH]-(d2:Drug {name: $drug2})
RETURN d1.name, d2.name, type(r) AS rel_type, r.effect, r.mechanism

-- 模板6：疾病并发症
MATCH (d1:Disease {name: $disease})-[:COMPLICATION]->(d2:Disease)
RETURN d2.name AS complication

-- 模板7：检查项目查询
MATCH (d:Disease {name: $disease})-[:NEED_CHECK]->(exam:Examination)
RETURN exam.name, exam.description

-- 模板8：药品不良反应
MATCH (drug:Drug {name: $drug})-[:CAUSES]->(effect:SideEffect)
RETURN effect.name, effect.severity

-- 模板9：人群用药
MATCH (drug:Drug {name: $drug})-[:APPLICABLE_TO]->(pop:Population)
RETURN pop.name, drug.dosage_adjustment

-- 模板10：食物建议
MATCH (d:Disease {name: $disease})-[r:DO_EAT|NOT_EAT]->(food:Food)
RETURN type(r) AS advice, food.name, food.reason
```

### 3.3 Text2Cypher安全沙箱

## 4. RAG检索增强生成（医疗领域）

### 4.1 医疗RAG架构最佳实践

基于2024-2025年最新研究，医疗领域RAG的核心优化方向：

| 方向 | 技术 | 效果 |
|------|------|------|
| 多步检索推理 | RaR（Radiology Retrieval & Reasoning） | 诊断准确率67%→75% |
| 图RAG + 知识图谱 | GraphRAG（Cypher RAG + Vector RAG） | 幻觉变异度降低53.94% |
| 合理性引导检索 | RAG²（Rationale-Guided RAG） | 准确率提升6.1% |
| 迭代验证精炼 | VeReaFine | 多跳推理错误减少26% |
| 文档嵌入增强 | DEEB-RAG | 准确率提升2.3% |
| 本地LLM+RAG | Llama 3.2-11B + RAG | 幻觉率0% vs 基线8% |

### 4.2 本项目的RAG流水线设计

```
用户问题 → NLU实体/意图解析
         → Text2Cypher图谱查询（结构化知识）
         → Milvus向量检索（非结构化文档）
         → Prompt组装（系统指令+图谱结果+向量文档+用户问题）
         → LLM生成答案
         → 置信度评估（Faithfulness+Context Relevance）
         → (<95% → 触发安全兜底; ≥95% → 通过)
         → 答案校验（实体一致性+来源标注+安全性+完整性）
         → 追加医疗免责声明
         → 返回最终答案
```

### 4.3 置信度量化公式

根据详细设计文档：

$$S_{final} = 0.2 \cdot \text{Similarity}_{Milvus} + 0.2 \cdot \text{Relation}_{Neo4j} + 0.4 \cdot \text{Faithfulness}_{Ragas} + 0.2 \cdot \text{AnswerRelevance}_{Ragas}$$

当 $S_{final} < 0.95$ 时触发兜底话术。

## 5. Milvus向量数据库

### 5.1 核心能力

- 十亿级向量毫秒级ANN检索
- HNSW索引（平衡检索速度与精度）
- pymilvus.asyncio异步客户端
- 云原生水平扩展（计算与存储分离）

### 5.2 医疗领域最佳实践

**操作顺序（官方推荐）**：
```
embed → create collection → insert → index → load → search → LLM generate
```

**检索策略**：混合检索（dense dense vectors + sparse BM25）

- 参考项目 medical-rag 的混合检索召回率达 **91.32%**（vs 纯dense 70.12%、纯BM25 61.08%）
- 使用pkuseg医学分词 + 医学停用词表

**Embedding模型推荐**：
- `bge-m3`（中文效果优秀）
- `nomic-embed-text-v1.5`
- `sentence-transformers/all-MiniLM-L12-v2`

### 5.3 双擎同步机制（Neo4j ↔ Milvus）

当图数据库写入事务成功提交后，FastAPI应用层自动在后台启动`SyncEmbeddingPipeline`：
1. 对对应实体的最新描述文本重新计算Embedding
2. 异步更新至Milvus（upsert操作）
3. 保证两套知识底座的强一致性

## 6. 药物相互作用检测

### 6.1 三级严重性分级

| 级别 | 标签 | 颜色 | 关系类型 | 动作 |
|------|------|------|---------|------|
| 一级 | 禁忌联用 | 红色 | CONTRAINDICATED_WITH | 强遮罩阻断对话框 |
| 二级 | 谨慎联用 | 橙色 | CAUTION_WITH | 橙色卡片提示 |
| 三级 | 潜在注意 | 黄色 | INTERACTS_WITH | 黄色说明卡片 |

### 6.2 核心Cypher查询：多药交叉比对

```cypher
MATCH (d1:Drug)-[r:CONTRAINDICATED_WITH|CAUTION_WITH|INTERACTS_WITH]-(d2:Drug)
WHERE d1.name IN $drugs AND d2.name IN $drugs AND id(d1) < id(d2)
RETURN d1.name AS drug_a, d2.name AS drug_b, type(r) AS rel_type,
       r.effect AS effect, r.mechanism AS mechanism
```

### 6.3 间接相互作用检测（多跳推理）

```cypher
MATCH path = (d1:Drug {name: $drug_a})-[:INTERACTS_WITH*1..3]-(d2:Drug {name: $drug_b})
RETURN path
```

## 7. RAG评估框架（Ragas/TruLens）

### 7.1 Ragas核心指标

| 指标 | 说明 | 计算方式 |
|------|------|---------|
| **Faithfulness（忠实度）** | 答案是否基于上下文 | 将答案拆分为claims，计算能从context中推断的比例 |
| **Context Precision** | 检索块的相关性排序 | 相关块是否排在不相关块之前 |
| **Context Recall** | 上下文与标准答案的覆盖率 | 每个ground truth句子是否能归因到检索上下文 |
| **Answer Relevancy** | 答案与问题的匹配度 | 从答案生成合成问题，计算cosine相似度 |

### 7.2 TruLens RAG Triad

| 指标 | 关注点 | 方法 |
|------|--------|------|
| **Context Relevance** | 检索器 | 每个检索块是否与query相关 |
| **Groundedness / Faithfulness** | 生成器 | 逐句验证响应是否被上下文支持 |
| **Answer Relevance** | 端到端 | 最终响应是否完整回答了用户问题 |

### 7.3 医学QA专用评估：ASTRID框架

针对临床对话场景的增强指标：

| ASTRID指标 | 说明 |
|-----------|------|
| **Conversational Faithfulness (CF)** | 区分acknowledgements/questions/informative语句，仅对informative语句验证忠实度 |
| **Refusal Accuracy (RA)** | 系统是否正确拒绝回答无信息问题（安全关键） |
| **Context Relevance (CR)** | 二元标注，强调临床信息完整性 |

## 8. 知识图谱可视化（D3.js/ECharts）

### 8.1 本项目选型方案

### 8.2 双引擎对比

| 维度 | D3.js | ECharts |

| 交互性 | 手动但无限制 | 内置（focus/roam/highlight） |
| 最佳用途 | 复杂定制化知识图谱 | 标准实体关系看板 |

### 8.3 本项目的彩色编码规范

## 11. 医疗数据安全与合规

### 11.1 三级等保核心要求

根据《医疗卫生机构网络安全管理办法》：

| 合规维度 | 要求 |
|---------|------|
| **等级测评** | 三级及以上系统每年至少一次 |
| **数据加密** | 传输加密（HTTPS/TLS 1.2+）、存储加密 |
| **匿名化/去标识化** | 科研与共享场景须对敏感信息做匿名哈希 |
| **数据分类分级** | 每年梳理一次，按重要性分级保护 |
| **密码应用** | 同步规划、同步建设、同步运行 |

### 11.2 本项目的安全设计实现

根据概要设计文档：

1. **匿名哈希**：用户ID全部转换为单向不可逆加盐哈希（Salted Hash），彻底切断物理身份关联
2. **密码存储**：bcrypt哈希加盐
3. **JWT认证**：短时效令牌（患者2小时/管理员1小时）
4. **二次鉴权**：敏感操作需管理员再次输入操作密码
5. **字段级加密**：患者身份、咨询记录等敏感数据AES字段级加密
6. **HTTPS全程**：TLS 1.2/1.3
7. **知识版本化**：所有知识更新保留历史版本与操作人员
8. **数据库每日备份**：加密存储至内网安全区
9. **医疗免责声明**：所有问答结果页统一展示

## 12. FastAPI + Neo4j 异步集成

### 12.1 py2neo → 官方驱动的迁移

**关键决策**：详细设计文档明确指出，py2neo已停止维护且同步阻塞，统一采用官方**Neo4j Python Driver (neo4j≥5.0)**。

### 12.2 异步DAO实现模式

```python
from neo4j import AsyncGraphDatabase

class Neo4jAsyncDAO:
    def __init__(self, uri: str, user: str, auth_token: str):
        self.driver = AsyncGraphDatabase.driver(
            uri, auth=(user, auth_token),
            max_connection_pool_size=100
        )

    async def execute_cypher_safely(self, cypher_query: str, params: dict):
        # 1. 安全沙箱检测
        forbidden = ["create", "delete", "set", "merge", "remove", "drop"]
        if any(kw in cypher_query.lower() for kw in forbidden):
            raise PermissionError("Unsafe Cypher statement blocked.")

        # 2. 只读事务执行
        async with self.driver.session(database="neo4j") as session:
            async def run_query(tx):
                limit_query = f"{cypher_query} LIMIT 200"
                result = await tx.run(limit_query, **params)
                return [dict(record) async for record in result]

            return await session.execute_read(run_query)
```
